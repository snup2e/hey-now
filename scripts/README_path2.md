# Path 2 — 1호선 실차 녹음 가이드

성균관대 → 구로 구간(13역)을 STM32 보드로 직접 녹음해서 데이터셋을 만드는 과정입니다.
폰 마이크는 안 씁니다 (실제 추론할 때의 마이크 도메인과 달라지면 학습이 의미 없음).

## 한 줄 요약

1. 보드(ICS43434 마이크 결선됨) USB로 노트북에 연결
2. `python scripts/path2_capture_ui.py --port <COMx>` → 뜨는 다이얼로그에서 **등교 / 하교** 선택
3. 트립 동안 안내방송이 시작될 때마다 **스페이스바** — 상단 dBFS 레벨미터로 마이크 입력 모니터링
4. 왕복이면 종점에서 **↻ 방향 전환** 버튼 누르고 계속 녹음
5. 도착해서 종료 → `python scripts/path2_slice.py` → 클립·메타데이터 자동 생성

## 0. 준비물

- STM32 NUCLEO-F411RE (펌웨어는 보드에 이미 플래시됨)
- ICS-43434 I2S 마이크 브레이크아웃
- **F-F 점퍼선 6가닥** (M-F + 빵판 조합은 실차에서 진동에 약함 — 아래 결선 참조)
- 마스킹 테이프, 양면테이프, 폼보드(또는 두꺼운 종이) — 보드 고정용
- **휴지 한 조각 또는 솜** — 마이크 윈드스크린용
- 노트북 + 마이크로 USB 케이블 (NUCLEO ST-LINK 포트용)
- Python 3.10+ : `pip install pyserial`

## 1. 결선 (F-F 직결, 빵판 X)

빵판이 가방 안에서 흔들리면 점퍼가 미세하게 빠져 데이터가 망가집니다 (LSB가 0x00/0xFF만 나오는 클리핑 패턴). F-F 점퍼로 **마이크 ↔ NUCLEO 모르포 핀 직결**이 안정적.

| 마이크 핀 | NUCLEO 핀 | 모르포 위치 |
|---|---|---|
| VCC | 3V3 | CN7-16 |
| GND | GND | CN10-9 |
| SEL | GND | CN10-20 (또는 마이크 위에서 GND핀과 짧은 점프) |
| BCLK | PB13 | CN10-30 |
| LRCL (WS) | PB12 | CN10-16 |
| DOUT (SD) | PB15 | CN10-26 |

총 F-F 6가닥. **F-F 암 소켓이 헐겁다면** 핀세트로 옆구리 살짝 눌러서 조이기.

### 물리 보호

- 마이크 보드 + NUCLEO 둘 다 **같은 폼보드/딱딱한 카드에 양면테이프로 고정** — 상대 움직임 방지
- 6가닥 점퍼를 **마스킹테이프로 한 다발**로 묶기 — 한 가닥이라도 단독으로 흔들리지 않게
- **마이크 포트(작은 동그란 구멍) 위에 휴지 한 조각** 가볍게 덮기 — 바람/핸들링 공기압 차단
  - 음파(안내방송)는 통과, 직접 공기 흐름은 차단
  - **박스/통/두꺼운 천은 ❌** — 공진 + 도메인 미스매치로 학습 망침
- 마이크 포트는 **위쪽(천장 스피커 방향)** 으로 향하게
- 가방 안에 넣을 땐 책 사이에 끼워 흔들림 방지

## 2. 사전 검증 (반드시 — 매 트립 출발 전)

가방에서 꺼냈을 때 점퍼가 살짝 어긋날 수 있으니, **매번 조용한 실내에서 RMS 확인 후 출발**:

```powershell
python scripts/path2_capture_ui.py --port COM<x>
```

다이얼로그 통과하면 캡처 창에서 **10초간 가만히 둔 채** 레벨미터 관찰:

| 결과 | 의미 | 대응 |
|---|---|---|
| RMS 20~50 / -50 dBFS 이하 / 초록색 끝부분 | 정상 ✓ | 트립 출발 |
| RMS 100~500 | 환경 노이즈 약간 큼 — 노트북 팬 등 | 조용한 곳으로 이동 후 재시도 |
| RMS 1000+ / -10 dBFS 위 / 노란/빨간 영역 | **클리핑** | 점퍼 5가닥 다 뺐다 다시 꽂기, 마이크 포트 막힘 확인 |
| 박수쳤을 때 RMS가 안 튐 | 마이크 안 동작 | SD/BCLK/WS 점퍼 확인, 안 되면 보드/마이크 교체 |

박수쳤을 때 RMS가 1000+ 로 튀고 다시 조용해지면 ✓. 이거 통과해야 진짜 녹음 시작.

## 3. 녹음 시작 전

보드를 노트북에 USB 연결하면 가상 COM 포트가 생깁니다.

- **Windows**: 장치관리자 → 포트 (COM & LPT) → `STMicroelectronics Virtual COM Port (COMx)` 확인. 보통 COM3~COM9.
- **macOS/Linux**: `ls /dev/tty.usbmodem*` 또는 `ls /dev/ttyACM*`

```powershell
python scripts/path2_capture_ui.py --port COM5
```

작은 다이얼로그가 먼저 뜸:
- **📚 등교 (구로 → 성균관대)** — 집에서 학교로 가는 트립
- **🏫 하교 (성균관대 → 구로)** — 학교에서 집으로 가는 트립

선택하면 본 캡처 창이 열림. **첫 안내방송 전에 띄워두기.** 왕복 트립이면 종점에서 ↻ 방향 전환 버튼으로 방향 바꾸고 계속 녹음 가능 — 하나의 audio.wav에 두 구간이 들어가고 marks.json에 segments 두 개로 저장됨.

## 4. 트립 중

- 안내방송 "이번 역은 ___" 시작 들리면 **스페이스바**
- 노란색 ▶ 표시가 다음 역으로 자동 이동
- 헷갈리거나 놓쳤으면 → 해당 역 버튼을 마우스로 클릭 (재마크도 동일하게 덮어쓰기)
- 마크가 살짝 늦거나 빠르게 눌려도 OK — 슬라이서가 ±4~12초 마진으로 자름
- **틈틈이 레벨미터 확인** — 안내방송 안 나올 때도 빨간/노란 영역에 가 있으면 클리핑 진행 중, 그 구간은 학습 못 씀

**팁**: 환승역(금정·금천구청·구로 등)은 안내방송이 길어서 "이번 역은"이 ~4초쯤 늦게 시작합니다. 들릴 때 누르면 됨.

## 5. 종료

- 마지막 역 도착 → 우측 하단 **종료 & 저장** (또는 창 X)
- 파일 두 개가 저장됨:
  - `data/raw/line1_live/<trip_id>/audio.wav` — 트립 통째 녹음
  - `data/raw/line1_live/<trip_id>/marks.json` — 역별 마크 타임스탬프
- `<trip_id>` = `YYYYMMDD_HHMM_<등교|하교>` (예: `20260524_0742_등교`)
- marks.json은 `segments[]` 배열 — 왕복이면 두 segment, 단방향이면 하나

## 6. 클립 + 메타데이터 생성

녹음 끝나고 노트북에서:

```bash
python scripts/path2_slice.py            # 새로 추가된 트립만 처리
python scripts/path2_slice.py --trip 20260524_0742_등교   # 특정 트립만
python scripts/path2_slice.py --pre 5 --post 14            # 윈도우 조정
```

생성물:
- `data/processed/line1_clips/<trip_id>/<역>_<seq>.wav` — 16초 클립 (마크 -4s ~ +12s)
- `data/path2_metadata.csv` — clip_file / station / trip_id / direction / mark_offset_sec / clip_sec / sample_rate

같은 트립을 다시 슬라이스하면 메타데이터 행이 덮어쓰기됩니다 (clip_file 키 기준).

## 7. 노트북 없이 보드 검증 (mock 모드)

보드/펌웨어가 아직 없어도 UI 동작은 확인 가능:

```bash
python scripts/path2_capture_ui.py --mock-wav data/processed/wav/성균관대.wav
```

mock 모드는 입력 wav를 무한 반복 재생하면서 실시간처럼 스트림합니다. UI 인터랙션, 저장 포맷, 슬라이서까지 전부 보드 없이 검증 가능.

## 트러블슈팅

| 증상 | 원인/대응 |
|---|---|
| 창은 뜨는데 elapsed 시계가 0:00에서 안 움직임 | USB 포트 오인식. 다른 COM 포트로 재시도, 보드 펌웨어가 ST-Link VCP 921600으로 송신 중인지 확인 |
| 녹음이 너무 빠르게/느리게 들림 | `--sr` 값 mismatch. 펌웨어가 보내는 샘플레이트와 일치시켜야 함 |
| **레벨미터가 평상시 -10 dBFS 위 (노란/빨간)** | **클리핑 — 데이터 못 씀**. 점퍼 헐겁거나 마이크 포트가 막힌 상태. 즉시 트립 중단 후 결선 점검 |
| 박수쳐도 RMS 변화 없음 | I2S 데이터선(특히 SD=PB15) 단선. F-F 다시 꽂기 |
| WAV 들어보면 노이즈만 들리고 안내방송 없음 | 같이 LSB가 0x00/0xFF만 나오면 마이크 결선 불안정 — 사전 검증(섹션 2) 통과 못 한 케이스 |
| 마크를 다 놓쳤음 | 트립 폐기 말고 일단 audio.wav 보관 → 나중에 Audacity로 수동 마킹 후 marks.json 직접 작성 가능 (스키마는 기존 파일 참조) |
| 파일이 손상되거나 wav 헤더가 깨짐 | Recorder가 정상 stop을 못 부른 경우. 데이터 자체는 raw PCM이라 살릴 수 있음 — 별도 복구 도구 필요 |
